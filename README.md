## My fav tech books so far :)

| Book                                                        | Hot Keys                                                                                                    | Why Read It?                                                                                                                                                                                                                                |
| ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A Philosophy of Software Design                             | Complexity management, deep modules, information hiding, reducing cognitive load                            | One of the best books on software design fundamentals. It teaches how to make systems easier to understand and maintain by reducing complexity rather than merely organizing it. Particularly valuable for senior engineers and architects. |
| Domain Modeling Made Functional                             | Functional DDD, algebraic data types, domain modeling, making illegal states unrepresentable                | Shows how to model business domains precisely using functional programming concepts. Even if you don't use F#, it dramatically improves how you think about domain boundaries and business rules.                                           |
| Software Architecture: The Hard Parts                       | Distributed systems trade-offs, architecture decisions, consistency, scalability, evolutionary architecture | Focuses on the difficult decisions architects face in real systems. Instead of teaching patterns, it teaches how to reason about trade-offs. Excellent for engineers moving into architecture roles.                                        |
| Patterns, Principles, and Practices of Domain-Driven Design | Strategic DDD, bounded contexts, aggregates, CQRS, event sourcing                                           | A practical bridge between DDD theory and implementation. Provides many concrete examples and architectural guidance for complex enterprise systems.                                                                                        |
| Domain-Driven Design                                        | Ubiquitous language, bounded contexts, aggregates, strategic design                                         | The original DDD book. Dense and sometimes difficult, but it fundamentally changed how software teams think about complex business domains. Essential reading for anyone serious about enterprise software design.                          |
| Building Microservices                                      | Service decomposition, deployment, resilience, observability, distributed systems                           | One of the most practical introductions to microservices. Covers not just architecture but also operational concerns such as monitoring, testing, and deployment.                                                                           |
| Joel on Software                                            | Software craftsmanship, product development, engineering culture, practical wisdom                          | A collection of essays that teaches timeless lessons about software engineering, teams, hiring, and product development. Easy to read and surprisingly relevant decades later.                                                              |
| Refactoring: Improving the Design of Existing Code          | Code smells, refactoring techniques, incremental improvement, maintainability                               | Probably the most influential book on improving existing code safely. It teaches systematic ways to evolve software without breaking behavior. Every professional developer benefits from it.                                               |
| The Pragmatic Programmer                                    | Engineering mindset, responsibility, continuous learning, practical techniques                              | Less about coding techniques and more about becoming an effective software professional. The advice applies across languages, frameworks, and decades of industry change.                                                                   |
